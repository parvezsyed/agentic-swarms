The OWA Architecture

The Problem

Single-agent coding assistants are great for demos. But when you try to ship real software with them, you hit three walls:

WALL 1

Quality

One agent writing and reviewing its own code is a conflict of interest. It will rubber-stamp its own mistakes — the same blind spots that wrote the bug review it.

WALL 2

Cost

Agentic workflows burn tokens on context bloat, cloud CI round-trips, and irrelevant tool schemas shipped on every API call. A 200-task plan isn't an asset — it's a liability.

WALL 3

Failure Modes

Silent transport timeouts, stale cache false-greens, directory isolation breaches, asynchronous racing. These are misdiagnosed as "model failures" but are actually pipeline bugs.

The insight: The expensive model's best use isn't writing more code — it's deciding what not to build. Spend more time cutting the plan than generating it.

The OWA Architecture

Assign agents opposing incentives. One plans and dispatches. One writes code. One tries to break it. A fourth verifies integration at milestone boundaries. None trust each other.

ORCHESTRATOR

The Planner

Flagship model · $0.50–3/call

Highest reasoning. Maintains global state. Judges whether code is safe to merge. Does not edit code. Accept, reject, or dispatch a fix.

WORKER

The Builder

Mid-tier model · per-token

Polite, accurate code generation. Mechanical tool use. Writes code, runs tests locally, returns diffs + test results on an isolated branch.

ADVERSARY

The Breaker

Free / subscription · $0/call

Zero incentive to be polite. Reads the Worker's code, writes aggressive failing tests, runs them. Reports breaks or "confirmed-safe." Catches what a 2am human reviewer rubber-stamps.

VERIFIER

The Integrator

Flagship · 1× per block

Fresh context. Runs once per block — not per subtask. Cross-subtask integration assurance: verifies subtask A's output is correctly consumed by subtask B's input.

Why quality goes UP with more agents: The Adversary has no reason to be polite. It writes failing tests on purpose. Catches that a human reviewer would miss at 2am get caught by an agent whose entire identity is "find the break."

The OWA Loop

For each subtask, run this loop. Per-subtask context stays tiny — ~5KB total. The entire block runs in ONE orchestrator session.

1

Dispatch Worker Orchestrator

Orchestrator dispatches Worker on an isolated worktree/branch. Worker writes code, runs tests locally, commits. Outputs a structured summary: status, files changed, acceptance criteria met/unmet.

2

Run Local Gate Orchestrator

Build + test + race + lint + security-scan — run in PARALLEL (see Phase 5). Clear all tool caches first. Wall clock collapses to the slowest single component (~1 min).

3

Gate Failed? → Fix Worker Worker

If gate fails, dispatch a fix Worker on the same branch. Repeat from step 2. No silent fallback — if the Orchestrator model is down, halt and surface to a human.

4

Dispatch Adversary Adversary

Adversary reads the code, writes aggressive failing tests, runs them, reports breaks (or "confirmed-safe"). It has ONLY terminal + file — it doesn't need a browser to break your code.

5

Adversary Broke It? → Fix Worker Worker

If Adversary breaks the code, dispatch a fix Worker (same branch), repeat from step 2. The Orchestrator merges ONLY after the Adversary confirms safe.

6

Merge Locally Orchestrator

No PR, no push, no CI per subtask. Merge is local. Write an OWA record (worker summary, adversary report, fix history, merge commit, gate result) to a file — not inline in context.

7

Prune & Next Orchestrator

Prune the worktree and branch. Move to the next subtask. At block completion (all subtasks merged): push once, create issues/PRs as a batch, run CI once.

Context efficiency: Worker JSON ~500 bytes · Gate result ~200 bytes · Adversary report ~1KB · OWA record ~2KB (written to file). Total per subtask: ~5KB. If you're above 10KB, something is bloating it.

Cost Optimization Levers

Six phases, each cutting cost or wall clock. Pull them in order — each compounds the last.

PHASE 0

Strategic Scoping

Write your PRD on the strongest reasoning model. Ask it to decompose. Bounce through a different model to challenge — not approve. Force a rigid P0 MVP. Kill the rest. If you have 200 tasks, you failed Phase 0.

→ 5-15 P0 subtasks, not 200
PHASE 1

Model-to-Role Allocation

Match model nature to the job. Flagship for judgment (Orchestrator). Mid-tier for code gen (Worker). Free/subscription for breaking (Adversary). Flagship once per block (Verifier). No silent fallback — fail loudly.

→ $0/call for the Adversary
PHASE 3

Local-First Execution

No PRs per subtask. No push per subtask. No CI per subtask. Do everything locally. Push once at block-end. macOS Actions runners bill at 10× — self-hosting eliminated a 3,000-min quota exhaustion.

→ ~50% wall clock, ~65% cost
PHASE 4

Context Pruning

Every loaded tool, skill, and MCP schema is tokens you pay for on every call. Disable vision, video, TTS, browser, social tools for a backend build. Give the Adversary ONLY terminal + file. An unpruned orchestrator ships 20KB of irrelevant schema per call.

→ ~250K tokens saved per block
PHASE 5

Parallel Gates

Run build · test · race · lint · security-scan concurrently, not sequentially. Wall clock collapses from 3-5 min to ~1 min (the slowest single component). Split Docker e2e into 4-5 batches with unique network IDs.

→ 3-5× faster gates
PHASE 6

Block-End Verification

Run ONE Verifier pass at block-end — not per subtask. In our audit, a per-subtask Verifier caught 0 novel issues across 9 subtasks. It's theater that costs ~89% more API calls for zero value. Move it to block-end, fresh context.

→ ~89% fewer Verifier calls

Operational Pitfalls

These are misdiagnosed as "model failures" but are actually pipeline bugs. Click to expand each.

1. Transport timeouts vs agent failures ▶

A long-running Worker (10-30 min) may be silently killed by a 60-second transport-layer timeout, leaving the Orchestrator hanging. Symptom: the Worker silently disappears mid-task. Fix: use a transport that survives long runs (tmux sessions, or a runner with configurable timeout aligned to max task duration).

2. Directory isolation breaches ▶

A Worker told to work in an isolated branch will cd back to the main repo if the main repo path is named anywhere in the prompt. Naming a location is an implicit instruction to navigate there. Fix: never name the main repo path in the Worker prompt — give it the worktree path only.

3. Stale state validations ▶

A passing lint or test gate is invalid if it relies on cached state from a previous run. The gate "passes" on stale results and hides real issues. Fix: force cache invalidation at the start of every gate — delete the linter cache, test cache, and build artifact cache.

4. Asynchronous racing ▶

If a fix task is injected after an integration failure, an auto-dispatcher may promote the next feature task in parallel. The fix and the next feature race; both touch the same files; one wins, one corrupts. Fix: require explicit synchronous blocks before initiating corrective branches.

5. Self-reported "CI green" is a claim ▶

Agents commit locally and may forget to push. A closure comment saying "CI green" on a red remote is a real thing that happened. Fix: always verify on the remote branch state, not on the agent's self-report.

6. Per-subtask Verifier is theater ▶

A Verifier run on every subtask catches what the Adversary already caught, or noise. It costs ~89% more API calls for zero novel value. Fix: Verifier runs once per block, at block-end, in a fresh context, focused on cross-subtask integration.

7. Mega-session context burn ▶

Running all subtasks in one Orchestrator session without pruning causes context to grow to 100K+ tokens; the model re-processes the entire history on every call. This is the single largest token-waste event in a build. Fix: prune before each block; split across sessions at natural breakpoints if context still grows.

8. Orchestrator edits code ▶

If the orchestrator has a file-patch tool, it will use it under pressure, bypassing the Adversary and the gate. Fix: strip the tool from the orchestrator profile at the tool-schema level. Prompts are suggestions; tool availability is a guarantee.

9. No-fallback violation ▶

If the Orchestrator model is down and the pipeline silently falls back to a cheaper model, every judgment call (merge, scope, break assessment) degrades. Fix: hard-stop the pipeline on any assigned-model outage. Surface to a human. Never substitute.

10. Duplicate CI gates ▶

If you run the same gate locally AND via CI on a self-hosted runner that IS your dev machine, you're paying the same compute twice plus the CI round-trip. Fix: in local-first mode, the local gate IS the CI. Push once at block-end; let CI run once as a safety net, not per subtask.

Post-Build Audit

An agent's closure comment ("CI green, all tests pass") is a claim, not evidence. Verify against ground truth before declaring the block done. Click each item to check it off.

✓
Fix commits are present on the REMOTE, not just local. git log origin/main must show the fix.
✓
CI is green on the remote main branch. Do not trust the closure comment — check the actual CI status.
✓
E2E tests that CI skips (e.g., Docker-gated suites) actually pass locally on the merged code.
✓
Any security suppressions (CVE ignores, lint exceptions) have factually accurate reasoning. "No fix available" is wrong if a fix exists.
✓
The CI workflow actually runs the current block's gate, not a stale first-block gate.
✓
Adversary and Verifier profiles have ONLY terminal + file — nothing else.
✓
Pruning restored after block completion — skills/toolsets re-enabled for the next block.
✓
Fast path defined for trivial changes (no Adversary/Verifier) — don't burn feature-level costs on one-line bumps.
Reopen if skipped: After a block completes, audit every subtask issue for skipped work — CVE resolutions, e2e tests, adversary tests. Investigate solutions on the internet before closing. Verify e2e tests actually ran before closing issues.

Key Principles

01

Cut, don't generate

The expensive model's best use is deciding what NOT to build. Spend more time cutting the plan than generating it.

02

Opposing incentives raise quality

Adding agents with opposing incentives makes quality go UP. The Adversary has no reason to be polite.

03

Orchestrator doesn't edit

Strip the file-patch tool from its profile. Every fix routes through a Worker and passes through the Adversary and the gate.

04

Cloud round-trip is a tax

If your CI hardware is your dev machine, running gates locally and pushing at block-end saves 50% wall clock and 65% cost.

05

Context is a budget

Every loaded tool, skill, and MCP schema is tokens you pay for on every call. Prune ruthlessly.

06

Theater catches nothing

A per-subtask Verifier caught 0 novel issues across 9 subtasks. Move it to block-end; keep the Adversary per-subtask.

07

Claims ≠ evidence

An agent's closure comment is a claim, not evidence. Always verify against the remote repository state.

08

OWA is overkill for trivial work

Maintain a fast path for one-line config bumps and doc typos. Don't burn feature-level costs for zero assurance.
